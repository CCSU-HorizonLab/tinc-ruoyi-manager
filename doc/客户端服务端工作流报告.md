# TincNet Manager — 客户端与服务端工作流报告

> 作者：sun | 日期：2026-05-31

---

## 概述

本系统涉及 **两套客户端**：

| 客户端 | 技术栈 | 角色 |
|--------|--------|------|
| **Web 前端** | Vue 2 + Element UI | 管理员通过浏览器进行服务器、内网、节点的 CRUD 管理 |
| **C++ Qt 客户端** | C++ / Qt | 终端用户的 Tinc VPN 节点程序，负责密钥生成、公钥上传、建立 VPN 连接 |

两者与 **Java Spring Boot 服务端** 的交互分为以下四个阶段。

---

## 阶段一：基础设施搭建（Web 前端 → 服务端）

### Step 1 — 服务器纳管

```
Web 前端 ── POST /manger/mangeServer ──▶ 服务端
```

**前端操作**：填写服务器名称、公网 IP、网段范围（起始/终止）、端口范围（起始/终止）

**服务端处理** (`MangeServerController.add`)：
- 校验 IP 格式（正则：`xxx.xxx.xxx.xxx`）
- 校验网段格式（正则：`xxx.xxx.xxx` 或 `xxx.xxx`）
- 写入 `mange_server` 表

---

### Step 2 — 创建 Tinc 内网（核心初始化）

```
Web 前端 ── POST /TincNetworkMange/TincNetworkMange ──▶ 服务端
```

**前端操作**：选择接入服务器 → 填写网络名称、网段、端口

**服务端处理** (`TincNetworkMangeServiceImpl.insertTincNetworkMange`)：

```
① 数据库入库
   networkStatus = "正常运行中"

② 反查真实 IP
   根据 serverName → 查询 mange_server 表 → 获取真实公网 IP

③ 初始化目录结构
   TincConfigUtils.initNetworkEnv(netName)
   → /etc/tinc/<netName>/
   → /etc/tinc/<netName>/hosts/

④ 生成 RSA 4096 密钥对
   RsaUtils.generateKeys()
   → BouncyCastle 纯正 PKCS#1 格式
   → 公钥头：-----BEGIN RSA PUBLIC KEY-----
   → 私钥头：-----BEGIN RSA PRIVATE KEY-----

⑤ 生成 tinc.conf
   createTincConf(netName, "server_master", "")
   Name = server_master
   Interface = tinc0
   Mode = switch
   ⚠️ 不生成 ConnectTo（中心服务器仅被动监听）

⑥ 生成网卡启停脚本
   createTincUpAndDown(netName, segment + ".1")
   ├── tinc-up        (Linux: ifconfig $INTERFACE x.x.x.1 netmask 255.255.255.0)
   ├── tinc-down      (Linux: ifconfig $INTERFACE down)
   ├── tinc-up.bat    (Windows: netsh interface ipv4 set address ...)
   └── tinc-down.bat  (Windows: netsh interface ipv4 set address source=dhcp)

⑦ 生成 hosts 文件（中心服务器公钥）
   createHostFile(netName, "server_master", subnet, realPublicIp, publicKey)
   → /etc/tinc/<netName>/hosts/server_master
   Address = <真实公网IP>
   Subnet = <网段>.1/32
   <RSA 公钥>

⑧ 写入私钥
   createPrivateKey(netName, privateKey)
   → /etc/tinc/<netName>/rsa_key.priv
```

**文件落盘结构**：

```
/etc/tinc/<netName>/                    （生产环境）
D:/tinc/<netName>/                      （开发环境）
  ├── tinc.conf                         # 网络配置
  ├── tinc-up                           # Linux 启动脚本
  ├── tinc-down                         # Linux 停止脚本
  ├── tinc-up.bat                       # Windows 启动脚本
  ├── tinc-down.bat                     # Windows 停止脚本
  ├── hosts/
  │   └── server_master                 # 中心服务器公钥文件
  └── rsa_key.priv                      # 中心服务器私钥
```

**事务保护**：若文件生成过程中任何步骤失败，`@Transactional(rollbackFor = Exception.class)` 自动回滚数据库记录。

## 阶段二：节点注册（Web 前端 → 服务端）

### Step 3 — 添加 Tinc 节点

```
Web 前端 ── POST /node_mange/node_mange ──▶ 服务端
```

**前端操作**：
- 级联选择：接入服务器 → 所属内网
- 填写：节点名称、内网 IP、密码
- `userName` 由 `$store.getters.name` 自动填充当前登录用户
- `networkIp` 根据所选内网的 `segment` 字段自动拼接 IP 前缀

**服务端处理** (`TincNodeMangeServiceImpl.insertTincNodeMange`)：

```
① 仅节点元信息入库
   创建时间 = 当前时间
   nodeStatus = 0（未配置）
   status = 0（初始状态）

② ⚠️ 不生成密钥
   密钥生成职责已从服务端转移至 C++ Qt 客户端
   日志记录："节点 [xxx] 基础信息创建成功，等待客户端上传公钥..."
```

## 阶段三：客户端下载安装包（C++ Qt 客户端 → 服务端）

### Step 4 — 下载节点配置安装包

```
C++ Qt 客户端 ── POST /node_mange/node_mange/download/{id} ──▶ 服务端
```

**服务端处理** (`TincNodeMangeController.download`)：

```
① 根据 id 查询节点信息
   获取 networkName, nodeName

② 拼装 ZIP 文件路径
   <BasePath>/zips/<netName>_<nodeName>.zip

③ 检查文件是否存在
   不存在 → 抛异常 "安装包文件不存在，请尝试删除节点后重新创建"

④ 设置响应头
   Content-Type: application/octet-stream

⑤ 流式写出 ZIP 文件
   FileUtils.writeBytes(filePath, response.getOutputStream())
```

**ZIP 包内容** (由 `ZipUtils.createZip` 打包)：

```
<netName>_<nodeName>.zip
  ├── tinc.conf                    # 网络配置
  ├── tinc-up                      # Linux 启动脚本
  ├── tinc-down                    # Linux 停止脚本
  ├── tinc-up.bat                  # Windows 启动脚本
  ├── tinc-down.bat                # Windows 停止脚本
  ├── hosts/
  │   └── server_master            # 中心服务器公钥（用于建立信任）
  └── rsa_key.priv                 # 节点私钥
```

**C++ Qt 客户端拿到 ZIP 后的本地操作**：

```
① 解压 ZIP 到本地 Tinc 安装目录
② 本地自行生成 RSA 密钥对
③ 用解压出的配置文件启动 Tinc 服务
④ 准备将本地生成的公钥上传至服务端
```

## 阶段四：公钥上传与配置完成（C++ Qt 客户端 → 服务端）

### Step 5 — 客户端上传公钥（核心联动环节）

```
C++ Qt 客户端 ── PUT /node_mange/node_mange ──▶ 服务端
请求体：{ id: xxx, password: "<含 PEM 公钥的文本>" }
```

**服务端处理** (`TincNodeMangeServiceImpl.updateTincNodeMange`)：

```
① 判断场景
   if (password 包含 "-----BEGIN" && password 包含 "PUBLIC KEY-----")
      → 公钥上传场景
   else
      → 普通字段更新（直接透传 Mapper）

② 【正则精准提取】公钥上传场景的处理：
   
   正则表达式：
   (?s)(-----BEGIN(?: RSA)? PUBLIC KEY-----.*?-----END(?: RSA)? PUBLIC KEY-----)
   
   从客户端传来的脏公钥文本中精准抠出纯净 PEM 块

③ 查询旧节点信息
   获取 netName, nodeName, networkIp（用于覆写 hosts 文件）

④ 【覆写 hosts 文件】
   TincConfigUtils.createHostFile(netName, nodeName, nodeIp, cleanPubKey)
   → /etc/tinc/<netName>/hosts/<nodeName>
   → 完全覆写，杜绝 Subnet 条目反复叠加（"叠罗汉"问题）

⑤ 【更新业务状态】
   setPassword(null)   // 公钥绝不存入数据库
   setStatus("已配置")  // 仅更新配置状态
   Mapper 执行 UPDATE
```

**关键设计**：`password` 字段在数据库操作中被双重复用：

| 场景 | password 字段内容 | 处理方式 |
|------|-------------------|----------|
| 公钥上传 | 含 PEM 公钥块文本 | 正则提取 → 写文件 → 设为 null → 更新状态 |
| 普通编辑 | 普通密码字符串 | 直接写入 DB |

## 内网编辑时的配置同步

当管理员通过 Web 前端修改内网信息时（如更换接入服务器、修改网段），服务端需要同步更新磁盘上的 Tinc 配置文件：

```
Web 前端 ── PUT /TincNetworkMange/TincNetworkMange ──▶ 服务端

服务端处理 (TincNetworkMangeServiceImpl.updateTincNetworkMange)：

1. 查出修改前的老数据（获取 netName）
2. 执行数据库 UPDATE
3. 读取旧 hosts/server_master 文件
4. 用正则提取原有公钥 PEM 块（保留密钥不变）
5. 根据新 serverName 反查真实公网 IP
6. 用新 Subnet + 新 IP + 老公钥 覆写 hosts/server_master
7. 重写 tinc-up/tinc-down 脚本（绑定新 .1 IP）
8. 若文件操作失败 → @Transactional 回滚数据库
```

## 关键设计要点汇总

| 设计点 | 详细说明 |
|--------|----------|
| **密钥不入库** | RSA 公私钥只存在于文件系统（`/etc/tinc/`），数据库仅存节点元信息和配置状态字段 |
| **PKCS#1 纯正格式** | 使用 BouncyCastle 底层 ASN.1 拼装 `RSAPublicKey`，生成 `-----BEGIN RSA PUBLIC KEY-----` 表头，100% 兼容 Tinc |
| **公钥精准提取** | 正则 `(?s)-----BEGIN(?: RSA)? PUBLIC KEY-----.*?-----END(?: RSA)? PUBLIC KEY-----` 匹配，容忍客户端上传的含噪声文本 |
| **hosts 文件覆写** | 客户端上传公钥时直接完全覆写，不进行追加，防止 Subnet 条目反复叠加（"叠罗汉"问题） |
| **密码字段复用** | `password` 在"公钥上传"场景下承载 PEM 公钥文本，在"普通更新"场景下为普通密码，由正则自动区分 |
| **Win/Linux 双路径** | 开发环境 `D:/tinc/`，生产环境 `/etc/tinc/`，通过 `System.getProperty("os.name")` 自动适配 |
| **防呆设计** | 中心服务器 `server_master` 的 `tinc.conf` 中不生成 `ConnectTo` 指令，仅作为被动监听者 |
| **事务保护** | `@Transactional(rollbackFor = Exception.class)` 确保数据库操作与文件操作的一致性 |
| **跨平台脚本** | 同时生成 Linux Shell (`tinc-up/down`) 和 Windows Batch (`tinc-up/down.bat`) 脚本 |
| **UTF-8 纯净写入** | 使用 NIO `Files.write()` 字节流直写，强行剔除 `\r\n`，杜绝 Windows CRLF 污染配置文件 |
| **模式** | 所有 Tinc 网络统一使用 `Mode = switch`（二层交换机模式） |

## 涉及的核心文件

| 文件 | 职责 |
|------|------|
| `ruoyi-common/.../utils/RsaUtils.java` | BouncyCastle 纯正 PKCS#1 RSA 密钥生成 |
| `ruoyi-common/.../utils/TincConfigUtils.java` | Tinc 配置文件生成与写入（tinc.conf/hosts/脚本/私钥） |
| `ruoyi-common/.../utils/ZipUtils.java` | 客户端安装包 ZIP 打包（保持 hosts/ 目录结构） |
| `ruoyi-common/.../utils/ShellUtils.java` | Linux Shell 命令执行（Windows 下 Mock） |
| `ruoyi-system/.../tinc_network/service/impl/TincNetworkMangeServiceImpl.java` | 内网创建/编辑的核心业务逻辑 |
| `ruoyi-system/.../tinc_node/service/impl/TincNodeMangeServiceImpl.java` | 节点公钥提取与 hosts 覆写逻辑 |
| `ruoyi-admin/.../controller/TincNodeMangeController.java` | 节点 CRUD + ZIP 下载接口 |
| `ruoyi-admin/.../controller/TincNetworkMangeController.java` | 内网 CRUD 接口 |
| `ruoyi-admin/.../controller/MangeServerController.java` | 服务器 CRUD 接口 |
| `ruoyi-ui/src/views/node_mange/node_mange/index.vue` | 前端节点管理页面（级联选择 + IP 自动填充） |
